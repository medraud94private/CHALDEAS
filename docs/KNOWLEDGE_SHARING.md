# 기술 지식 공유 가이드

## 왜 지식을 공유해야 하는가?

- **기억 보조**: 나중에 같은 문제를 만났을 때 빠르게 해결
- **팀 협업**: 팀원들이 동일한 문제에 시간 낭비하지 않음
- **커뮤니티 기여**: 다른 개발자들에게 도움
- **개인 브랜딩**: 전문성 어필, 커리어 발전

---

## 지식 공유 방법 비교

### 1. 프로젝트 내 문서화

| 위치 | 용도 | 예시 |
|------|------|------|
| `README.md` | 프로젝트 소개, 빠른 시작 | 설치 방법, 기본 사용법 |
| `docs/` | 상세 기술 문서 | 아키텍처, API, 배포 가이드 |
| `CONTRIBUTING.md` | 기여 가이드 | 코딩 컨벤션, PR 규칙 |
| `CHANGELOG.md` | 변경 이력 | 버전별 변경사항 |
| 코드 주석 | 복잡한 로직 설명 | 왜 이렇게 했는지 |

**장점**: 코드와 함께 버전 관리, 접근성 좋음
**단점**: 외부 공유 어려움

### 2. 위키/노션

| 플랫폼 | 특징 | 추천 상황 |
|--------|------|----------|
| **Notion** | 협업, 데이터베이스, 템플릿 | 팀 내부 지식 베이스 |
| **Confluence** | 엔터프라이즈, Jira 연동 | 회사 프로젝트 |
| **GitHub Wiki** | 저장소 연동, 무료 | 오픈소스 프로젝트 |
| **GitBook** | 깔끔한 UI, 버전 관리 | 공개 문서 |

### 3. 기술 블로그

| 플랫폼 | 특징 | SEO |
|--------|------|-----|
| **velog** | 한국 개발자 커뮤니티, 마크다운 | 좋음 |
| **Medium** | 글로벌, 유료화 가능 | 매우 좋음 |
| **dev.to** | 글로벌 개발자 커뮤니티 | 좋음 |
| **Hashnode** | 커스텀 도메인 무료 | 좋음 |
| **개인 블로그** | 완전한 제어 | 직접 관리 |

### 4. 코드 스니펫

| 방법 | 용도 |
|------|------|
| **GitHub Gist** | 짧은 코드, 설정 파일 공유 |
| **CodePen** | 프론트엔드 데모 |
| **CodeSandbox** | 전체 프로젝트 데모 |
| **StackBlitz** | 빠른 프로토타입 |

## 어떤 방법을 언제 사용할까?

```
┌─────────────────────────────────────────────────────────────┐
│                    지식 공유 의사결정 트리                     │
│                                                              │
│  이 지식이 프로젝트에 특화되어 있나?                           │
│        │                                                     │
│        ├── Yes ──→ docs/ 폴더에 문서화                       │
│        │           └── 팀 공유 필요? ──→ Notion 복사         │
│        │                                                     │
│        └── No (범용적 지식)                                  │
│                 │                                            │
│                 ├── 짧은 팁/스니펫 ──→ GitHub Gist           │
│                 │                                            │
│                 ├── 튜토리얼/가이드 ──→ 기술 블로그           │
│                 │                                            │
│                 └── 시리즈/연재 ──→ 블로그 + 목차 정리        │
└─────────────────────────────────────────────────────────────┘
```

## 효과적인 기술 문서 작성법

### 1. 문서 구조 템플릿

```markdown
# 제목 (무엇을 다루는가)

## 개요
- 이 문서의 목적
- 대상 독자
- 사전 지식

## 배경/문제
- 왜 이 주제가 중요한가
- 어떤 문제를 해결하는가

## 해결 방법
- 단계별 설명
- 코드 예시
- 주의사항

## 결과/검증
- 어떻게 확인하는가
- 예상 결과

## 참고 자료
- 공식 문서 링크
- 관련 글
```

### 2. 좋은 문서의 특징

| 특징 | 설명 | 예시 |
|------|------|------|
| **명확한 제목** | 검색 가능, 내용 예측 | "Cloud Run으로 FastAPI 배포하기" |
| **실행 가능** | 복사해서 바로 실행 | 완전한 코드 블록 |
| **맥락 설명** | 왜 이렇게 하는지 | "이 방법을 선택한 이유는..." |
| **트러블슈팅** | 흔한 문제와 해결책 | "권한 오류 시..." |
| **최신 정보** | 버전, 날짜 명시 | "2024년 1월 기준" |

### 3. 코드 블록 작성 팁

```bash
# 나쁜 예: 컨텍스트 없음
gcloud run deploy

# 좋은 예: 전체 명령어와 설명
# Cloud Run에 백엔드 서비스 배포
# - 이미지: gcr.io/프로젝트/서비스명
# - 리전: asia-northeast3 (서울)
gcloud run deploy chaldeas-backend \
  --image gcr.io/PROJECT_ID/chaldeas-backend \
  --region asia-northeast3 \
  --platform managed \
  --allow-unauthenticated
```

## 지식 공유 체크리스트

### 문서 작성 전

- [ ] 대상 독자는 누구인가? (초보/중급/고급)
- [ ] 이 지식의 유효 기간은? (버전 의존성)
- [ ] 비슷한 문서가 이미 있는가? (중복 방지)

### 문서 작성 중

- [ ] 코드는 복사해서 바로 실행 가능한가?
- [ ] 스크린샷/다이어그램이 필요한가?
- [ ] 흔한 오류와 해결책을 포함했는가?

### 문서 작성 후

- [ ] 다른 사람이 따라했을 때 동작하는가?
- [ ] 관련 문서에 링크를 추가했는가?
- [ ] 날짜/버전을 명시했는가?

## 결론

1. **프로젝트 지식** → `docs/` 폴더에 마크다운으로
2. **팀 협업** → Notion에 복사/링크
3. **범용 지식** → 기술 블로그에 정리
4. **코드 스니펫** → GitHub Gist
